## Single Responsibility principle
Нужна только одна причина изменения кода
![[Pasted image 20230803131925.png]]
то есть для каждой сущности - одна причина изменения кода
Преймущества: 
1) код сфокусирован только на одной задаче
2) повторное использования
3) снижение рисков побочных эффектов

## Open closed principle
Программа должна быть открыта для расширения, но закрыты для модификации.
![[Pasted image 20230803133205.png]]
Очевидно нельзя закрыться на 100%, нужно попытаться угадать расширение.
Следствия из этого принципа:
1) не использовать глобальные принципы
2) все переменные члены - закрытые (private)
3) RTTI - опасно

## Liskov Substituation Principle
Проблема возникает тогда, когда мы делаем такую штуку: нужно сначала вызвать какой то метод, а потом наш
Пример:
пингвин умеет плавать
жираф - нет,
добавим в их общий базовый класс метов плавать, который бросает exception - это плохо
решение: нужно придумать другую иерархию

## Interface Segregation Principle
нужно сводить к миниуму взаимодействия между компонентами
клиенты не должны видеть интерфейс с которым не взаимодействуют

Преймущества: 
1) переиспользование(модульность)
2) скрытие реализации
3) снижение зависимости между друг другом
